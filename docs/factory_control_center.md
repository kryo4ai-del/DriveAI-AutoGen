# Factory Control Center

Last Updated: 2026-03-10

---

## What It Is

The Factory Control Center is a lightweight browser-based dashboard for the AI App Factory.
It provides read-only visibility into all factory stores — ideas, projects, specs, compliance, orchestration plans, and more.

Built with Python + Streamlit. Designed to run on the existing Ubuntu/Docker server alongside other services.

### Activity Feed

The Activity Feed (`activity_feed.py`) aggregates events from all factory stores into a normalized chronological feed. Each event has:

| Field | Description |
|---|---|
| event_type | Human-readable label (e.g. "Idea Created", "Compliance Warning", "Plan Approved") |
| source_store | Which store it came from (e.g. "ideas", "compliance") |
| ref_id | Item ID (e.g. "IDEA-001", "LEGAL-002") |
| title | Item title or description |
| project | Linked project or "—" |
| severity | Priority/severity/risk level or "—" |
| timestamp | Date for chronological sorting |

Event types are derived from each item's current status:

| Source | Example Events |
|---|---|
| Ideas | Idea Created, Idea Classified, Idea Prioritized, Idea Blocked |
| Specs | Spec Drafted, Spec Approved, Spec In Progress, Spec Completed |
| Orchestration | Plan Created, Plan Approved, Plan Executing, Plan Completed |
| Opportunities | Opportunity Detected, Opportunity Evaluated, Opportunity Accepted |
| Watch Events | Watch Alert, Watch Acknowledged, Watch Resolved |
| Compliance | Compliance Warning, Compliance Reviewed, Compliance Blocker |
| Accessibility | Accessibility Warning, Accessibility Fixed |
| Content | Content Draft Created, Content Published |
| Bootstrap | Project Bootstrapped, Project Released |

The feed is **read-only** and **derived** — no separate event store is needed. Events are aggregated fresh on each page load from existing JSON stores.

The Overview page shows a preview of the latest 10 events. The dedicated Activity Feed page shows up to 100 events with filters for source, project, and severity.

### Self-Improving Factory Layer

The Factory Improvement Analyzer (`improvements/analyzer.py`) inspects all monitoring signals and generates improvement proposals. It is a lightweight, read-only analyzer — no background workers or databases.

**Signal Sources**:
- Watch events (SDK updates, security changes, model updates, unresolved critical events)
- Compliance findings (blockers, recurring topics, pending external reviews)
- Accessibility reports (critical issues, recurring issue types)
- Orchestration plans (blocked plans, stale drafts)
- Opportunities (high-relevance items awaiting evaluation)
- Agent memory (recurring implementation patterns, architecture gaps)

**Proposal Structure** (`improvements/improvement_proposals.json`):

| Field | Description |
|---|---|
| proposal_id | Auto-incremented ID (IMP-001, IMP-002, ...) |
| title | Short description of the improvement |
| summary | Detailed explanation |
| category | security / performance / tooling / automation / model_update / architecture / sdk_update / compliance / accessibility / general |
| severity | critical / high / medium / low / info |
| affected_systems | List of affected stores or projects |
| recommended_action | What to do about it |
| detected_from | Which signal source triggered this |
| status | new / evaluated / accepted / in_progress / completed / rejected / deferred |
| created_at | ISO date |

**Running the Analyzer**:
```bash
cd /path/to/DriveAI-AutoGen
python -m improvements.analyzer
```

The analyzer is idempotent — it checks existing proposal titles before creating new ones to avoid duplicates.

### AI Trend Scanner

The AI Trend Scanner (`trends/scanner.py`) detects emerging AI opportunities from internal signals and automatically generates idea candidates for high-relevance trends.

**Signal Sources**:
- Watch events (model updates, SDK requirements, tooling changes, security updates)
- Opportunities (market-relevant opportunities mapped to trend categories)
- Agent memory (AI keyword patterns in decisions/architecture notes, recurring feature patterns in implementation notes)

**Trend Structure** (`trends/trend_store.json`):

| Field | Description |
|---|---|
| trend_id | Auto-incremented ID (TREND-001, TREND-002, ...) |
| title | Short description of the trend |
| category | model_release / developer_tooling / app_category / framework_update / platform_change / ai_capability / market_shift / automation_pattern / general |
| summary | Detailed explanation |
| relevance_score | 0.0–1.0 (computed from category weight + severity boost) |
| potential_app_categories | List of platforms (ios, web, android) |
| detected_from | Which signal source triggered this |
| status | detected / evaluated / idea_generated / dismissed / expired |

**Automatic Idea Generation**:
Trends with `relevance_score >= 0.70` automatically generate an idea in the factory idea store via `IdeaManager.add_idea()`. Generated ideas are prefixed with `[Auto]` and include the linked trend ID, platforms, and complexity estimate in their notes.

**Running the Scanner**:
```bash
cd /path/to/DriveAI-AutoGen
python -m trends.scanner
```

The scanner is idempotent — it checks existing trend titles and idea titles before creating new ones.

### Opportunity Radar

The Opportunity Radar (`radar/radar_manager.py`) is a lightweight external signal intake and evaluation layer. It collects product opportunity signals before they become trends, opportunities, or ideas.

**How it differs from Trends and Opportunities**:

| Layer | Purpose | Source |
|---|---|---|
| Radar | Raw external signals — "something interesting exists out there" | Product Hunt, HN, Reddit, newsletters, manual observations |
| Trends | Detected patterns from internal factory signals | Watch events, opportunities, agent memory, radar hits |
| Opportunities | Evaluated product ideas ready for action | Trends, watch events, manual discovery |

**Signal Flow**:
```
External World → Radar Hit → (score/evaluate) → Opportunity → Idea → Spec → Implementation
                     ↓
              Trend Scanner (picks up radar hits as signal source)
```

**Radar Source** (`radar/radar_sources.json`):

| Field | Description |
|---|---|
| source_id | Auto-incremented ID (RSRC-001, RSRC-002, ...) |
| name | Human-readable source name |
| category | product_hunt / app_store / hacker_news / reddit / twitter_x / newsletter / blog / research_paper / github_trending / ai_directory / marketplace / competitor / community / manual |
| url | Source URL or identifier |
| enabled | Whether the source is active |
| notes | Additional context |
| created_at | ISO date |

**Radar Hit** (`radar/radar_hits.json`):

| Field | Description |
|---|---|
| hit_id | Auto-incremented ID (RADAR-001, RADAR-002, ...) |
| source_id | Reference to RSRC-NNN |
| title | Short description of the signal |
| summary | Detailed explanation |
| category | new_product / feature_pattern / market_gap / pricing_model / tech_stack / user_pain_point / viral_concept / monetization / platform_shift / regulation / general |
| relevance_score | 0.0–1.0 (auto-scored from category weight + product/platform boosts) |
| potential_products | List of potential product ideas |
| potential_platforms | List of platforms (ios, web, android, etc.) |
| status | new / evaluated / promising / opportunity_created / dismissed / expired |
| detected_at | ISO date |

**Hit Lifecycle**:
```
new → evaluated → promising → opportunity_created
                            → dismissed
                            → expired
```

**Scoring**: Hits are auto-scored based on category weight (market_gap = 0.85, new_product = 0.80, etc.) plus boosts for multiple potential products/platforms.

**Promotable Hits**: Hits with status `evaluated` or `promising` and `relevance_score >= 0.70` are considered promotable — ready to become opportunities via OpportunityManager.

**Integration Points**:
- **Trend Scanner**: Scans radar hits as a signal source, creates trends from active hits
- **Daily Briefing**: Includes radar KPIs (sources, hits) and promotable hit actions
- **Control Center**: Dedicated Radar page + Radar section on Overview
- **OpportunityAgent**: Can consume radar hits to generate structured opportunity records

**Usage**:
```python
from radar.radar_manager import RadarManager
mgr = RadarManager()

# Add a source
mgr.add_source("Product Hunt", "product_hunt", url="https://producthunt.com")

# Add a hit
mgr.add_hit("AI-powered budget tracker", "new_product",
            source_id="RSRC-001", summary="Trending on PH, 500+ upvotes",
            relevance_score=0.8, potential_platforms=["ios", "web"])

# Score all new hits
mgr.auto_score_all_new()

# Find promotable hits
for hit in mgr.promotable():
    print(f"{hit['hit_id']}: {hit['title']} ({hit['relevance_score']:.0%})")
```

**V1 Design Principles**:
- No heavy crawler infrastructure
- No aggressive scraping or always-on workers
- Structure and pipeline first — external fetch logic is minimal/stub-friendly
- JSON file storage, no database
- Lightweight and server-friendly

### ResearchMemoryGraph

The ResearchMemoryGraph (`research_graph/graph_manager.py`) is a lightweight knowledge graph that captures relationships between all factory entities.

**Purpose**: Factory stores contain isolated records (ideas, trends, opportunities, etc.). The graph layer connects them so you can see *how* entities relate — which idea came from which trend, which compliance finding affects which project, which radar hit became which opportunity.

**Node Types** (13):
idea, project, opportunity, trend, radar_hit, spec, compliance, accessibility, strategy_report, improvement, watch_event, briefing, content

**Edge Types** (10):
| Edge | Meaning |
|---|---|
| derived_from | B was derived from A (e.g. idea from trend) |
| related_to | Topically related entities |
| promoted_to | A was promoted to B (e.g. radar hit → opportunity) |
| blocked_by | A is blocked by B |
| recommended_for | A is recommended for project B |
| affects | A affects B (e.g. compliance → project) |
| linked_to | Generic bidirectional link |
| generated_from | A was auto-generated from B |
| addresses | A addresses/fixes B |
| depends_on | A depends on B |

**Data Ingestion** (`research_graph/ingest.py`):
Reads all factory stores and builds nodes + edges based on cross-references:
- Ideas linked to projects (recommended_for)
- Ideas derived from trends (derived_from)
- Specs linked to ideas and projects
- Radar hits promoted to opportunities
- Compliance/accessibility findings affecting projects
- Improvements generated from watch events
- Strategy reports linked to featured opportunities/trends/risks
- Category-based trend↔opportunity relationships

**Running Ingestion**:
```bash
python -m research_graph.ingest
```
Idempotent — skips existing nodes/edges on re-run.

**Querying**:
```python
from research_graph.graph_manager import GraphManager
gm = GraphManager()

# Connected context for an entity
ctx = gm.connected_context("IDEA-001")
# → {entity, edges_out, edges_in, neighbors_by_type, total_connections}

# Most connected entities
top = gm.most_connected(10)

# Neighbors of an entity
neighbors = gm.neighbors_by_entity("TREND-003")

# Graph stats
stats = gm.stats()
```

### ModelRouter & AICostMonitor

The ModelRouter (`config/model_router.py`) selects the optimal AI model for each task, and the AICostMonitor (`costs/cost_manager.py`) tracks usage costs and enforces budgets.

**Model Routing Strategy**:

| Task Type | Default Model | Provider | Reason |
|---|---|---|---|
| classification | ollama/mistral | ollama | lightweight |
| summarization | ollama/mistral | ollama | lightweight |
| trend_analysis | ollama/mistral | ollama | pattern matching |
| scoring | ollama/mistral | ollama | numerical evaluation |
| briefing | ollama/mistral | ollama | summarization |
| planning | gpt-4o-mini | openai | requires reasoning |
| code_generation | gpt-4o | openai | advanced reasoning |
| code_review | gpt-4o-mini | openai | code understanding |
| architecture | gpt-4o | openai | deep reasoning |
| content_generation | gpt-4o-mini | openai | creativity |

Routes are configurable via `config/model_routing.json`. Each agent has a default task type mapping — the router uses this to auto-select the model.

**Cost Tracking** (`costs/cost_usage.json`):

| Field | Description |
|---|---|
| usage_id | Auto-incremented (COST-0001, ...) |
| agent_name | Which agent made the call |
| model_used | Model name (e.g. gpt-4o-mini) |
| task_type | Task category (e.g. code_generation) |
| prompt_tokens | Input tokens |
| completion_tokens | Output tokens |
| estimated_cost | Cost in USD |
| project | Which project this was for |
| timestamp | ISO timestamp |

**Budget Configuration** (`config/cost_budgets.json`):
```json
{
  "daily_budget": 5.00,
  "monthly_budget": 100.00,
  "alert_threshold_percent": 80
}
```

When budget thresholds are exceeded, alerts appear in:
- Control Center Overview
- AI Costs dashboard page
- Daily Briefing alerts section

**Usage**:
```python
from config.model_router import ModelRouter
from costs.cost_manager import CostManager

# Route a task
router = ModelRouter()
route = router.route("code_generation")
# → {"model": "gpt-4o", "provider": "openai", "reason": "requires advanced reasoning"}

# Route for a specific agent
model = router.get_model_for_agent("swift_developer")
# → "gpt-4o"

# Log usage
costs = CostManager()
costs.log_usage("swift_developer", "gpt-4o", "code_generation",
                prompt_tokens=2000, completion_tokens=1500,
                estimated_cost=router.estimate_cost("gpt-4o", 2000, 1500))

# Check budget
budget = costs.check_budget()
print(budget["alerts"])
```

**Integration with llm_config.py**:
```python
from config.llm_config import get_routed_llm_config

# Get AutoGen-compatible config using ModelRouter
config = get_routed_llm_config(agent_name="swift_developer")
# Automatically selects model, handles Ollama base_url for local models
```

## Read-Only Behavior

The Control Center is strictly read-only in v1:
- All JSON stores are mounted as read-only volumes in Docker (`ro` flag)
- StoreReader never writes to any file
- No database is used — data is read directly from JSON on every page load
- Changes made via Claude/CLI/pipeline are visible on next browser refresh

### Dashboard shows "Factory root not found"
- Check the `FACTORY_ROOT` env var or volume mount
- Verify the mounted path contains the factory repo (look for `main.py` or `config/agent_toggles.json`)
- Docker: `docker exec factory-control-center ls /data/main.py`

### Dashboard shows 0 for everything
- Stores may be empty (normal for a fresh factory)
- Check the "Data Store Health" expander on the Overview page
- Verify mount: `docker exec factory-control-center ls /data/factory/ideas/`

### Container keeps restarting
- Check logs: `docker logs factory-control-center --tail 50`
- Common cause: missing `curl` in image (needed for healthcheck) — fixed in Dockerfile
- Memory limit too low: increase `mem_limit` if needed (256 MB is usually enough)

### Pages load slowly
- Normal on first load (Streamlit cold start ~3-5s)
- Subsequent page switches are fast
- If persistently slow: check if JSON files are very large

### Updating after factory changes
- Changes from Claude/CLI/pipeline are picked up on next page load
- No restart needed — Streamlit reads JSON files fresh on each request
- For Docker: the volume mount is live, no rebuild needed for data changes
- For code changes: `docker compose up -d --build`
