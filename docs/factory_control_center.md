# Factory Control Center

Last Updated: 2026-03-10

---

## What It Is

The Factory Control Center is a lightweight browser-based dashboard for the AI App Factory.
It provides read-only visibility into all factory stores — ideas, projects, specs, compliance, orchestration plans, and more.

Built with Python + Streamlit. Designed to run on the existing Ubuntu/Docker server alongside other services.

### Activity Feed

The Activity Feed (`activity_feed.py`) aggregates events from all factory stores into a normalized chronological feed. Each event has:

| Field | Description |
|---|---|
| event_type | Human-readable label (e.g. "Idea Created", "Compliance Warning", "Plan Approved") |
| source_store | Which store it came from (e.g. "ideas", "compliance") |
| ref_id | Item ID (e.g. "IDEA-001", "LEGAL-002") |
| title | Item title or description |
| project | Linked project or "—" |
| severity | Priority/severity/risk level or "—" |
| timestamp | Date for chronological sorting |

Event types are derived from each item's current status:

| Source | Example Events |
|---|---|
| Ideas | Idea Created, Idea Classified, Idea Prioritized, Idea Blocked |
| Specs | Spec Drafted, Spec Approved, Spec In Progress, Spec Completed |
| Orchestration | Plan Created, Plan Approved, Plan Executing, Plan Completed |
| Opportunities | Opportunity Detected, Opportunity Evaluated, Opportunity Accepted |
| Watch Events | Watch Alert, Watch Acknowledged, Watch Resolved |
| Compliance | Compliance Warning, Compliance Reviewed, Compliance Blocker |
| Accessibility | Accessibility Warning, Accessibility Fixed |
| Content | Content Draft Created, Content Published |
| Bootstrap | Project Bootstrapped, Project Released |

The feed is **read-only** and **derived** — no separate event store is needed. Events are aggregated fresh on each page load from existing JSON stores.

The Overview page shows a preview of the latest 10 events. The dedicated Activity Feed page shows up to 100 events with filters for source, project, and severity.

### Self-Improving Factory Layer

The Factory Improvement Analyzer (`improvements/analyzer.py`) inspects all monitoring signals and generates improvement proposals. It is a lightweight, read-only analyzer — no background workers or databases.

**Signal Sources**:
- Watch events (SDK updates, security changes, model updates, unresolved critical events)
- Compliance findings (blockers, recurring topics, pending external reviews)
- Accessibility reports (critical issues, recurring issue types)
- Orchestration plans (blocked plans, stale drafts)
- Opportunities (high-relevance items awaiting evaluation)
- Agent memory (recurring implementation patterns, architecture gaps)

**Proposal Structure** (`improvements/improvement_proposals.json`):

| Field | Description |
|---|---|
| proposal_id | Auto-incremented ID (IMP-001, IMP-002, ...) |
| title | Short description of the improvement |
| summary | Detailed explanation |
| category | security / performance / tooling / automation / model_update / architecture / sdk_update / compliance / accessibility / general |
| severity | critical / high / medium / low / info |
| affected_systems | List of affected stores or projects |
| recommended_action | What to do about it |
| detected_from | Which signal source triggered this |
| status | new / evaluated / accepted / in_progress / completed / rejected / deferred |
| created_at | ISO date |

**Running the Analyzer**:
```bash
cd /path/to/DriveAI-AutoGen
python -m improvements.analyzer
```

The analyzer is idempotent — it checks existing proposal titles before creating new ones to avoid duplicates.

### AI Trend Scanner

The AI Trend Scanner (`trends/scanner.py`) detects emerging AI opportunities from internal signals and automatically generates idea candidates for high-relevance trends.

**Signal Sources**:
- Watch events (model updates, SDK requirements, tooling changes, security updates)
- Opportunities (market-relevant opportunities mapped to trend categories)
- Agent memory (AI keyword patterns in decisions/architecture notes, recurring feature patterns in implementation notes)

**Trend Structure** (`trends/trend_store.json`):

| Field | Description |
|---|---|
| trend_id | Auto-incremented ID (TREND-001, TREND-002, ...) |
| title | Short description of the trend |
| category | model_release / developer_tooling / app_category / framework_update / platform_change / ai_capability / market_shift / automation_pattern / general |
| summary | Detailed explanation |
| relevance_score | 0.0–1.0 (computed from category weight + severity boost) |
| potential_app_categories | List of platforms (ios, web, android) |
| detected_from | Which signal source triggered this |
| status | detected / evaluated / idea_generated / dismissed / expired |

**Automatic Idea Generation**:
Trends with `relevance_score >= 0.70` automatically generate an idea in the factory idea store via `IdeaManager.add_idea()`. Generated ideas are prefixed with `[Auto]` and include the linked trend ID, platforms, and complexity estimate in their notes.

**Running the Scanner**:
```bash
cd /path/to/DriveAI-AutoGen
python -m trends.scanner
```

The scanner is idempotent — it checks existing trend titles and idea titles before creating new ones.

### Daily Briefing Agent

The Daily Briefing Agent (`briefings/daily_briefing.py`) generates a professional executive morning summary from all factory data.

**Input Sources**: All 12+ factory stores — ideas, projects, specs, opportunities, trends, watch events, compliance, accessibility, orchestration, improvements, memory, agent toggles.

**Briefing Sections**:
- Executive Summary — high-level status and alert count
- Actions Today — prioritized action items (critical alerts first, then triage tasks)
- Alerts & Risks — all critical/high items across watch, compliance, orchestration, a11y, ideas
- Project Status — all registered projects with activity status
- Ideas in Inbox — unclassified ideas awaiting triage
- AI Trends — top trends by relevance score
- Opportunities — active opportunity pipeline
- Compliance & Legal — open compliance reports with risk levels
- Accessibility — open a11y issues
- Factory Improvements — active improvement proposals

**Output Formats**:
1. **JSON record** in `briefings/briefing_store.json` — full structured data for dashboard display
2. **HTML email** in `briefings/html/briefing_YYYY-MM-DD.html` — professional executive layout with KPI cards, color-coded severity badges, clean tables
3. **Google Sheets row** via `to_sheets_row()` — flat dict with all KPIs + summary + actions for daily log append

**Running the Briefing Agent**:
```bash
cd /path/to/DriveAI-AutoGen
python -m briefings.daily_briefing
```

The agent is idempotent per day — running it twice on the same day returns the existing briefing.

**Delivery Preparation**:
- HTML output is ready for email delivery (e.g. via n8n, Make, or SMTP)
- `to_sheets_row()` returns a flat dict for Google Sheets API append
- Briefing store records track delivery status (generated → delivered → archived)

## Running Locally

```bash
cd control_center
pip install -r requirements.txt
streamlit run app.py --server.port=8502
```

Opens at `http://localhost:8502`.

## Deploying on Server (Docker)

### Option 1: docker-compose (recommended)

```bash
# On the server, from the factory repo root
cd control_center

# Build and start
docker compose up -d --build

# Check status
docker compose ps
docker compose logs -f factory-control-center
```

The dashboard runs on port 8502. The factory repo is mounted read-only at `/data`.

### Option 2: Standalone docker build

```bash
cd control_center
docker build -t factory-control-center .
docker run -d \
  --name factory-control-center \
  -p 8502:8502 \
  -v /path/to/DriveAI-AutoGen:/data:ro \
  --memory=256m \
  --cpus=0.5 \
  --restart=unless-stopped \
  factory-control-center
```

### Exposing via Cloudflare Tunnel

Add a public hostname in the Cloudflare dashboard or cloudflared config:
- Service: `http://localhost:8502`
- Hostname: e.g. `factory.yourdomain.com`

## Read-Only Behavior

The Control Center is strictly read-only in v1:
- All JSON stores are mounted as read-only volumes in Docker (`ro` flag)
- StoreReader never writes to any file
- No database is used — data is read directly from JSON on every page load
- Changes made via Claude/CLI/pipeline are visible on next browser refresh

## Troubleshooting

### Dashboard shows "Factory root not found"
- Check the `FACTORY_ROOT` env var or volume mount
- Verify the mounted path contains the factory repo (look for `main.py` or `config/agent_toggles.json`)
- Docker: `docker exec factory-control-center ls /data/main.py`

### Dashboard shows 0 for everything
- Stores may be empty (normal for a fresh factory)
- Check the "Data Store Health" expander on the Overview page
- Verify mount: `docker exec factory-control-center ls /data/factory/ideas/`

### Container keeps restarting
- Check logs: `docker logs factory-control-center --tail 50`
- Common cause: missing `curl` in image (needed for healthcheck) — fixed in Dockerfile
- Memory limit too low: increase `mem_limit` if needed (256 MB is usually enough)

### Pages load slowly
- Normal on first load (Streamlit cold start ~3-5s)
- Subsequent page switches are fast
- If persistently slow: check if JSON files are very large

### Updating after factory changes
- Changes from Claude/CLI/pipeline are picked up on next page load
- No restart needed — Streamlit reads JSON files fresh on each request
- For Docker: the volume mount is live, no rebuild needed for data changes
- For code changes: `docker compose up -d --build`

## V1 Limitations

- Read-only (no editing factory stores from the dashboard)
- No authentication (rely on Cloudflare Access or network-level security)
- No real-time auto-refresh (manual browser refresh to see changes)
- No historical trends or charts (shows current state only)
